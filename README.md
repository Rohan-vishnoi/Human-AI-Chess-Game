# AI-Human-Chess-Game
A single-player chess game with a basic UI, hosted on Google Cloud. The user plays as either White or Black, while the Gemini LLM acts as the opponent's 'brain' to generate moves.

# Define
### Updated Phase 1: Define & Requirements

#### 1. UI Flow & Welcome Screen
* **Welcome Screen:** The initial view when a user loads the app.
    * **Player Name:** An input field to capture the user's name.
    * **Color Selection:** A toggle or buttons allowing the user to choose to play as White or Black pieces.
    * **Start Button:** Initiates the game, locks in the settings, and transitions to the game board.
* **Game Over Popup:** A modal that appears when the game ends, displaying the player's name, the outcome, and perhaps a "Play Again" button that routes back to the Welcome screen.

#### 2. Database & State Tracking
Since you want to store results but *not* persist the live board state, here is how we will handle it:
* **The Database:** Since we are deploying on GCP, **Firestore** (Google's NoSQL document database) is the perfect, lightweight choice.
* **Game Record:** When a user clicks "Start", we create a record in the database.
* **Status ENUM:** You suggested `IN_PROGRESS`, `WIN`, and `LOSE`. 
    * *Candid Reality Check:* Because this is chess, we absolutely need to add **`DRAW`** to this ENUM to account for stalemates, insufficient material, or the 50-move rule.
* **Refresh Behavior:** As you specified, the board state is strictly maintained in the frontend Angular application. If the user refreshes the page, the state is wiped, and they are sent back to the Welcome screen. (The previous database record will simply remain stuck as `IN_PROGRESS`, which is standard for lightweight games).

#### 3. Game Logic & Validation
* **Move Validation:** The frontend will strictly validate all moves before allowing them or sending them to the AI. This includes castling, en passant, and pawn promotion. (Using `chess.js` handles all of this automatically).
* **AI First Move:** If the user selects Black on the welcome screen, the frontend must immediately trigger the AI to make the first move as White before the user can interact with the board.

#### 4. Updated Technology Stack
* **Frontend:** Angular, `chess.js` (validation), `ngx-chess-board` (UI).
* **Backend / Serverless:** Google Cloud Functions (to hide the Gemini API key and talk to the DB).
* **Database:** **Firestore** (to store Player Name, Color, and Game Status).
* **Hosting:** Firebase Hosting or Google Cloud Storage + Cloud CDN.

# Low level and High level designs


### Phase 2: High-Level Design (HLD)

The HLD gives us the "bird's-eye view" of how the different pieces of your system talk to each other.

**1. Architecture Diagram (Data Flow)**
* **Client Side (Angular App):** Hosted on Firebase Hosting / GCP Storage. Manages the UI, user interactions, and local chess board state.
* **Backend (Cloud Functions):** Acts as a secure middleman. 
* **Database (Firestore):** Stores the game records.
* **AI Engine (Gemini API):** Processes the board state and returns the next move.

**2. Core Interactions**
* **Start Game:** Angular App $\rightarrow$ Cloud Function $\rightarrow$ creates a record in Firestore (Status: `IN_PROGRESS`).
* **Play Game:** Angular App $\rightarrow$ sends the current board state (FEN string) to Cloud Function $\rightarrow$ Cloud Function queries Gemini API $\rightarrow$ returns AI move to Angular App.
* **End Game:** Angular App detects Checkmate/Draw $\rightarrow$ Cloud Function $\rightarrow$ updates Firestore record (Status: `WIN`, `LOSE`, or `DRAW`).

### Phase 2: Low-Level Design (LLD)

The LLD breaks down the specific components, database schemas, and API payloads we need to build.

#### 1. Angular Application Structure
We will divide the frontend into distinct, manageable pieces.

* **Components:**
    * `WelcomeComponent`: Contains the input for Player Name, the White/Black selection toggle, and the "Start Game" button.
    * `ChessBoardComponent`: Renders the board (using `ngx-chess-board`), handles drag-and-drop, and triggers move validation.
    * `GameOverModalComponent`: The popup that shows the final result and a "Play Again" button.
* **Services:**
    * `GameStateService`: Keeps track of the `gameId` (returned from the DB) and current player info. If this is empty (e.g., on a page refresh), it immediately routes the user back to the Welcome screen.
    * `ChessLogicService`: A wrapper around `chess.js`. Generates the FEN strings, validates user moves, and checks for game-over conditions.
    * `ApiService`: Handles all HTTP calls to your Cloud Functions.

#### 2. Database Schema (Firestore)
Firestore is document-based. We will have a single collection called `games`. 
* **Collection:** `games`
    * `document_id` (Auto-generated by Firestore)
    * `playerName`: string (e.g., "Rahul")
    * `playerColor`: string ("WHITE" or "BLACK")
    * `status`: string ("IN_PROGRESS", "WIN", "LOSE", "DRAW")
    * `createdAt`: timestamp

#### 3. API Contracts (Cloud Functions)
Your Angular app will make requests to these endpoints.

**Endpoint 1: `POST /startGame`**
* **Request Payload:** `{ "playerName": "Rahul", "playerColor": "WHITE" }`
* **Action:** Creates a document in Firestore.
* **Response:** `{ "gameId": "xyz123abc" }`

**Endpoint 2: `POST /getAiMove`**
* **Request Payload:** `{ "fen": "rnbqkbnr/pppppppp/8/8/4P3/8/PPPP1PPP/RNBQKBNR b KQkq - 0 1", "aiColor": "BLACK" }`
* **Action:** Securely formats a prompt and sends it to Gemini.
    * *The System Prompt:* "You are a chess engine playing as [aiColor]. The current board state in FEN notation is [fen]. What is your next move? Reply strictly with the move in standard algebraic notation (e.g., Nf6, e5, O-O) and nothing else."
* **Response:** `{ "move": "e5" }`

**Endpoint 3: `PUT /endGame`**
* **Request Payload:** `{ "gameId": "xyz123abc", "status": "WIN" }`
* **Action:** Updates the corresponding document in Firestore.
* **Response:** `{ "success": true }`
